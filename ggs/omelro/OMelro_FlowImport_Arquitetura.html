O que vamos construir

O FlowImport transforma a gestão de importações num processo digital, automatizado e auditável. No centro está um Agente IA que lê emails e documentos, extrai os dados e prepara-os para validação.

  • 📥 Emails e documentos não estruturados (BL, faturas, certificados, packing lists)
  • 🔄 Extração assistida por IA, com validação humana
  • 🗂️ Cada contentor como objeto digital estruturado no GProd
  • 🔗 Enriquecido com tracking Maersk e dados financeiros do PHC

Enquadrar: hoje são 30–90 min por email de importação, introdução manual, erro elevado. O objetivo é < 10 min com taxa de erro < 1%.

Princípio de desenho

Pipeline orquestrado com etapas de IA e validação humana — não um agente autónomo que decide sozinho.

Porquê esta distinção?

A O Melro precisa de estar permanentemente pronta para IFS v8, SMETA e HACCP. Isso exige previsibilidade e auditabilidade, não autonomia.

  • ✅ Cada etapa é registada, reproduzível e validável
  • ✅ Nenhum dado entra em produção sem passar por regras objetivas
  • ✅ O humano mantém sempre o controlo final sobre o que é gravado

Este é o argumento-chave de confiança. Se o cliente perguntar “a IA decide sozinha?”, a resposta é não — é um pipeline controlado com checkpoints.

Visão geral do fluxo

Cinza = entrada · azul = passos com IA · âmbar = controlo · verde = plataforma. Tudo sobre uma camada de auditoria transversal.

Etapa 1 — Ingestão

  • ✅ Caixa de correio dedicada (IMAP ou Microsoft Graph), em polling ou webhook
  • ✅ Extração de anexos: PDF, imagens, Excel — e do corpo do email
  • ✅ Deteção de PDF digital (com texto) vs. digitalizado
  • ✅ Separação por página e normalização

O Claude lê PDF e imagem nativamente por visão — não é preciso um motor OCR separado para documentos digitalizados.

Etapa 2 — Classificação

Cada anexo é classificado pelo Claude Haiku — modelo rápido e económico, ideal para volume.

Tipo de documento Encaminhamento
Bill of Lading → extração BL
Fatura comercial → extração fatura
Certificado fitossanitário → extração certificado
Packing list → extração packing list
Outro / não relevante → revisão ou arquivo

✅ Triagem barata antes da extração detalhada — só se gasta o modelo mais caro onde faz falta.

Etapa 3 — Extração estruturada

O Claude Sonnet extrai os dados com um esquema JSON por tipo de documento, forçado via tool use: o modelo devolve estrutura validada, não texto livre.

{
  "type": "object",
  "properties": {
    "numero_bl":      { "type": ["string", "null"] },
    "navio":          { "type": ["string", "null"] },
    "porto_carga":    { "type": ["string", "null"] },
    "porto_descarga": { "type": ["string", "null"] },
    "contentores": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "numero":        { "type": "string" },
          "selo":          { "type": ["string", "null"] },
          "peso_bruto_kg": { "type": ["number", "null"] }
        },
        "required": ["numero"]
      }
    }
  },
  "required": ["numero_bl"]
}

Campos ausentes são null — o modelo não inventa para preencher lacunas.

Campos extraídos por documento

Bill of Lading Nº BL, navio/viagem, portos de carga e descarga, expedidor/consignatário, nº contentor + selo, volumes, peso bruto.

Fatura comercial Nº fatura, fornecedor, incoterm, moeda, linhas (produto, quantidade, preço, total), total, condições de pagamento.

Certificado fitossanitário Nº certificado, país/autoridade emissora, exportador, produto (nome botânico), tratamento, declaração, data.

Packing list Nº contentor, volumes, peso líquido/bruto, repartição por produto.

Controlo de alucinação

A preocupação habitual: “e se a IA inventa dados?” — Respondemos com três camadas concretas.

1 · Prompt restritivo

Instrução explícita para devolver null em campos ausentes e nunca inferir. O modelo não preenche lacunas.

2 · Validadores determinísticos

Código (não IA) verifica o output antes de qualquer gravação.

3 · Encaminhamento por confiança

Falha de validação ou campo crítico em falta → fila de revisão humana. Documentos limpos → confirmação ligeira.

Este é o slide que responde diretamente à dúvida de fiabilidade. A IA acelera, mas as regras objetivas e o humano é que garantem.

Validadores determinísticos — exemplos

  • ✅ Nº de contentor — validação do dígito de controlo (ISO 6346)
  • ✅ Datas — formato e intervalos plausíveis (ETD ≤ ETA)
  • ✅ Reconciliação de totais — soma das linhas = total da fatura (dentro de tolerância)
  • ✅ Coerência entre documentos — nº de contentor do BL = packing list
  • ✅ Excerto-fonte — o modelo pode devolver o trecho de origem de cada campo, para verificação rápida pelo operador

Qualquer regra que falhe sinaliza o campo e encaminha para revisão — a IA nunca contorna a validação.

Reconciliação e modelo de dados

Os documentos chegam soltos; o sistema liga-os ao processo certo pelas chaves de junção:

nº de contentor + nº de BL + fornecedor/PO

Se não existir processo, cria um novo. Correspondência aproximada do fornecedor contra os dados mestre.

Objetos do modelo de dados: ProcessoImportação · Contentor · Documento (tipado) · Fornecedor · LinhaCusto · Pagamento · EventoTracking

Integrações

🚢 Maersk — Track & Trace

Consulta por nº de contentor ou BL. Devolve eventos, navio, localização, ETA e milestones. Polling periódico atualiza o Mapa de Chegadas.

Depende de credenciais e documentação da Maersk.

💶 PHC — contexto financeiro

Leitura de encomendas de venda, faturas de compra e pagamentos.

Margem por processo = valor de venda − custo landed (fatura + frete + direitos).

Profundidade a validar no levantamento, conforme conectores do PHC.

Modelos e proteção de dados

Modelos Claude

  • ✅ Haiku — classificação e triagem (volume)
  • ✅ Sonnet — extração estruturada e raciocínio
  • ✅ Opus — opcional, casos de maior complexidade
  • ✅ Regime pay-per-use por token

RGPD

  • ✅ Inferência em região da UE (AWS Bedrock / Vertex)
  • ✅ Por defeito, a API não treina com os dados (Termos Comerciais)
  • ✅ DPA + Cláusulas Contratuais-Tipo
  • ✅ Opção Zero Data Retention para dados sensíveis

Importante: esta solução é cloud Claude em região UE — não é “100% local”. A proteção de dados vem da região UE + não-treino + DPA/ZDR + validação humana.

Custos de IA — cloud em região UE

O preço por token do modelo é igual à API direta; o extra da residência na UE é apenas ~10%.

Componente Custo adicional
Preço por token (modelo) 0% — igual à API direta
Endpoint UE (residência de dados) +10%
Egress, logging, KMS Marginal
Provisioned Throughput / Knowledge Bases Evitável — não usados

Ordem de grandeza

~$0.02/página · ~$0.10–0.15 por processo (~6 páginas) · +10% UE. Para ~500 processos/mês ≈ €55–95/mês. Mantêm-se os descontos: batch −50% (histórico), prompt caching até −90%.

Valores ilustrativos — fechados com a volumetria real no levantamento. Custos de terceiros contratados diretamente pela O Melro.

Mensagem-chave: o custo de IA é da ordem de dezenas de euros/mês à escala da O Melro, e o “extra UE” é só 10% — não muda a ordem de grandeza.

Infraestrutura

  • ✅ GProd — aplicação Blazor / ASP.NET Core (motor, base de dados, processos, permissões)
  • ✅ Serviço de extração como microserviço dedicado, desacoplado por fila de mensagens
  • ✅ Ingestão → fila → workers de extração → resultados para o GProd
  • ✅ Absorve picos (dezenas de emails de manhã) sem bloquear

Se a IA estiver indisponível, o core mantém-se operável (introdução manual) — a falha na IA suspende automações, não a plataforma.

Para o histórico, a Batch API processa o backlog a metade do custo, fora do tempo real.
