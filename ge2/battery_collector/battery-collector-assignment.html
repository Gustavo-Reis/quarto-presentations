1 Introduction

This assignment will introduce you to C++ programming and Unreal Engine 5 (UE5), including some of the Gameplay Framework classes. It will also show you some of the different ways to work with classes, functions, and variables that can help you with your project. This is just a jumping off point to get you started, but it is a mini game with both winning and losing. As you will see, the character can collect batteries which are spawned by a Spawn Volume above the level. Your power increases as you collect the batteries, which changes your material and your running speed. However, the GameMode is also draining your character over time. You can lose power completely. By the end of this assignment, you will have a number of classes including a base pickup, a battery pickup which fires lightning towards the character and gives some of its base power to the character. You will have an upgraded character which has a Collection Sphere to collect all of the batteries within a certain radius of the character. You will have a GameMode which drains the character’s power over time. The GameMode also defines what happens when you win and when you lose during gameplay. You will have a Spawn Volume that spawns any kind of pickup whether it is a Base Pickup, a Battery Pickup, or some other kind of pickup that you develop on your own. There is also the HUD, which is set up using UMG (Unreal Motion Graphics) and has the power bar and the text at the top of the screen.

So, what is the first thing our pickup needs? Most pickups have a visual representation within the level. Let’s set ours up to have a Static Mesh Component. We are going to make our Static Mesh Component private. This means we can only set up and alter the Static Mesh Component within the pickup code. No other code classes, nor Blueprints, nor anything else can directly access the Static Mesh Component.

UCLASS()
class BATTERYCOLLECTOR_API APickup : public AActor
{
    GENERATED_BODY()

public:
    // Sets default values for this actor's properties
    APickup();

protected:
    // Called when the game starts or when spawned
    virtual void BeginPlay() override;

public:
    // Called every frame
    virtual void Tick(float DeltaTime) override;

private:
    /** Static mesh to represent the pickup in the level */
    UStaticMeshComponent* PickupMesh;
};

This is going to be our “Static Mesh to represent the pickup in the level”. Just like we have the UCLASS() macro, we also have UPROPERTY() and UFUNCTION() macros. We are going to use the UPROPERTY() macro here. Components are basically regarded as variables, but we need to add some specific keywords in here.

private:
    /** Static mesh to represent the pickup in the level */
    UPROPERTY(VisibleAnywhere, BlueprintReadOnly, Category = "Pickup",
              meta = (allowprivateaccess = true))
    UStaticMeshComponent* PickupMesh;

BlueprintReadOnly means that if you place the pickup directly in the level or create a Blueprint based on the pickup, you can see this component exists. However, you can’t remove it or edit it in a way so we wouldn’t rely on the pickup mesh being there and being a component of our Actor. You can also put a category keyword on it. We are going to add a Pickup category. This means in the Details panel, we can look under the Pickup category and our mesh will show up. The last thing we are going to use is this meta keyword. There are a few of these meta keywords throughout sample code, some of the templates, and especially on components. This code is specific to components in that this is private. We need to be able to change properties about the mesh such as which Static Mesh is assigned, or what size it is, etc. from within the Blueprint or within the level.

This is private, but we may need to return the Static Mesh and alter it from another class. Maybe when the Character powers up, all of the pickups start glowing and we need to change the material on the Static Mesh. Let’s make an accessor function for that. This function will “Return the mesh for the pickup.”

public:
    // Called every frame
    virtual void Tick(float DeltaTime) override;

    /** Return the mesh for the pickup */
    FORCEINLINE UStaticMeshComponent* GetMesh() const { return PickupMesh; }

We are going to use a little bit of magic here with the FORCEINLINE keyword. We want to change the material on the mesh from another class, and this function will allow us to do that. We made the GetMesh function const because it is not allowed to modify our Mesh Component, it will just return it for us. Finally, let’s set up the component within our constructor so it is ready to go when the pickup is created:

APickup::APickup()
{

First, we will “Create the Static Mesh Component”. There is a helper that will do that for us called CreateDefaultSubobject. UStaticMeshComponent is the type of component we are making. We can also give it some display text. We also want to set that StaticMeshComponent as the Root Component. If we add other effects or another mesh later, they would all be based on the size and location of the PickupMesh. At this point, we can compile. Switch back to the editor. 4 Adding Variables and Functions

Now, we will add our activity code to the pickup. Go back to Visual Studio. We want all pickups to have a state of whether it is active, to have ways to report whether it is active, and also to change it. Let’s start with our variable for whether it is active. We are going to make this variable protected which means we can change it in this code class, another code class based on this one, or in a Blueprint. If we wanted to create a Blueprint based on pickup and then directly toggle the pickup behavior, we could do that.

protected:
    /** True when the pickup can be used, and false when the pickup is deactivated */
    bool bIsActive;

This will be “True when the pickup can be used, and false when pickup is deactivated”. Again, that is very straightforward. The variable is just a Boolean. It will be true when the pickup can be used, and false when the pickup can’t be used. We will go to the constructor where we set the default values.

APickup::APickup()
{
    // Set this actor to call Tick() every frame. You can turn this off to
    // improve performance if you don't need it.
    PrimaryActorTick.bCanEverTick = false;

    // All pickups start active
    bIsActive = true;

    // Create the static mesh component
    PickupMesh = CreateDefaultSubobject<UStaticMeshComponent>(TEXT("PickupMesh"));
    RootComponent = PickupMesh;
}

All pickups start active. If you had a specific type of pickup where you wanted it to be inactive until the character did something, you could make a Blueprint or a code class based on this variable and override it in the constructor or the Construction Script because that variable is protected. Again, because this variable is protected, it can only be changed by other types of pickups and not by the Character or the GameMode. We would need to make functions for that. Let’s go ahead and do that now. We will create a function to return if the variable is active or not.

/** Return whether or not the pickup is active */
UFUNCTION(BlueprintPure, Category = "Pickup")
bool IsActive();

The function to return will return a Boolean. Boolean functions should ask a question that has a true or false answer. Is it active? Like we used the UPROPERTY() macro before, we are going to use the UFUNCTION() macro on this function. We are going to use the keyword BlueprintPure, which will be explained shortly. Then, we will create a function to change whether or not the variable is active.

/** Allows other classes to safely change whether or not pickup is active */
UFUNCTION(BlueprintCallable, Category = "Pickup")
void SetActive(bool NewPickupState);

The SetActive is not going to return anything. It will be a void function. This is also going to be a UFunction. This function is going to be BlueprintCallable.

BlueprintPure and BlueprintCallable are both things defined in code, but can be called from Blueprints. Technically BlueprintPure functions can modify things, but in general they shouldn’t. BlueprintPure functions are run every time their output is used. A BlueprintPure function that pops something off a stack that is called twice, either with a loop or in 2 separate nodes, will pop 2 things off the stack. On the other hand, BlueprintCallable functions only run when the input execution pin is hit. BlueprintCallable functions save their results in the output pins. BlueprintCallable functions can be used for modifying calls. Expensive calculations that aren’t expected to change frequently are better in BlueprintCallable functions than BlueprintPure functions. Fast calculations that will frequently change are better to make in BlueprintPure functions, or by adding multiple Blueprint Callable nodes to your graph.

Neither of these two functions you just created do anything right now. Let’s write the definition in the Pickup.cpp source file.

// Returns active state
bool APickup::IsActive()
{
    return bIsActive;
}

// Changes active state
void APickup::SetActive(bool NewPickupState)
{
    bIsActive = NewPickupState;
}

We have a Boolean for IsActive. It is very simple. Setting it up with these public functions that manipulate a protected variable keeps things code safe. This is especially so if you have multiple people working on a project and they may not understand the rule setup in code, or if you have people trying to have different things interact together.

Our setter is also very straightforward. If you also wanted to print the new pickup state to the screen or change how the mesh looks depending on whether it is active, you can handle all of that in here. It is a little more complex than just changing bIsActive, so you wouldn’t want to change bIsActive without those associated pieces of code.

The body of this function is going to be a simple switch statement. This is going to switch on the value of NewState. We have our different case statements which are all based on our 4 states of the EBatteryPlayState. If we have won the game, we want the spawn volumes to be inactive. void ABatteryCollectorGameMode::HandleNewState(EBatteryPlayState NewState)
{
    switch (NewState)
    {
    // If the game is playing
    case EBatteryPlayState::EPlaying:
    {
        // spawn volumes active
        for (ASpawnVolume* Volume : SpawnVolumeActors)
        {
            Volume->SetSpawningActive(true);
        }
    }
    break;

    // If we've won the game
    case EBatteryPlayState::EWon:
    {
        // spawn volumes inactive
        for (ASpawnVolume* Volume : SpawnVolumeActors)
        {
            Volume->SetSpawningActive(false);
        }
    }
    break;

    // If we've lost the game
    case EBatteryPlayState::EGameOver:
    {
        // spawn volumes inactive
        for (ASpawnVolume* Volume : SpawnVolumeActors)
        {
            Volume->SetSpawningActive(false);
        }

        // block player input
        APlayerController* PlayerController = UGameplayStatics::GetPlayerController(this, 0);
        if (PlayerController)
        {
            PlayerController->SetCinematicMode(true, false, false, true, true);
        }

        // ragdoll the character
        ACharacter* MyCharacter = UGameplayStatics::GetPlayerCharacter(this, 0);
        if (MyCharacter)
        {
            MyCharacter->GetMesh()->SetSimulatePhysics(true);
            MyCharacter->GetMovementComponent()->MovementState.bCanJump = false;
        }
    }
    break;

    // Unknown/default state
    default:
    case EBatteryPlayState::EUnknown:
    {
        // do nothing
    }
    break;
    }
}
